# ArcadeBloom 站点完备性与 SEO 完善建议（不含用户体系）

范围：仅从游戏站自身的“信息完备性 + SEO”出发，不考虑社交媒体传播；站点不做登录/用户管理。

原则：优先保证“可抓取/可索引/可复用”，再做内容扩展与性能优化。

---

## 0. 优先级总览（建议按 P0 → P2 落地）

**P0（直接影响收录与排名）**
- 每个游戏提供独立、稳定、可静态抓取的落地页 URL（不要只依赖 `game-detail.html?slug=...` 的客户端注入元数据）。
- canonical 与真实 URL 完全一致（目前部分页面 canonical 无 `.html`，容易造成重复收录/权重分散）。
- sitemap 只收录规范（canonical）URL；必要时做 sitemap 分片与索引；`lastmod` 可维护。
- 每个游戏页输出完整元信息：`title`、`meta description`、`canonical`、（如有多语言）`hreflang`。
- 结构化数据：站点级 `WebSite` + `SearchAction`（需与真实搜索 URL 对齐）；游戏页 `VideoGame`/`Game` + `BreadcrumbList`。

**P1（提升站内质量与长尾覆盖）**
- 分类/标签聚合页（可分页、可排序），形成清晰层级与内链网络。
- 搜索体验产品化：独立搜索页 URL 可分享；对“高频搜索词/专题”做静态落地页。
- 内容页补齐：榜单、专题集合、玩法指南、FAQ/术语表（不依赖社媒也能获取自然流量）。

**P2（性能、体验与长期维护）**
- Core Web Vitals：减少阻塞资源、图片优化、缓存策略、降低 JS 体积。
- 可访问性与语义化：标题层级、`alt`、键盘可用、焦点管理、对比度。
- 监控与自动化：Search Console、死链/重复 canonical 检测、sitemap 自动生成与校验。

## 1. 站点完备性（不做用户体系也能“完整”）

### 1.1 必备页面与导航闭环
- `404.html`：提供搜索框、热门分类/热门游戏入口，避免用户直接离站。
- 分类落地页：例如 `category/puzzle/`，支持分页与排序（热门/最新/评分）。
- 标签落地页：只保留核心标签（避免生成大量薄内容页），同样支持分页。
- 规则明确的聚合页：`/new/`、`/popular/`、`/featured/`，并在页面注明计算口径与更新时间。
- “提交游戏”页：增加收录规则（版权/可用性/内容要求），减少低质提交。

### 1.2 游戏落地页内容（模板化 + 避免薄内容）
每个游戏页建议至少包含：
- 清晰的游戏简介（避免全站同模板句式），并在首屏可见。
- 玩法说明与操作方式（键盘/触屏），建议可折叠但默认可抓取。
- 截图/封面与基础信息（类型、标签、发布时间/来源如可靠）。
- 相关游戏推荐：同分类/同标签/同操作方式三路推荐，保证每页 5–10 条高质量内链。
- 若为外链游戏：标注“外部站点打开”，并提供站内替代推荐，降低跳出。

### 1.3 站内搜索的“产品完备性”
- 提供独立 `search.html`（或 `/search/`）页面，URL 可分享：`?q=...&category=...`。
- 对“高频搜索词/强意图关键词”做静态专题页（例如“best puzzle games”集合页），既提升体验也利于 SEO。

## 2. 技术 SEO（关键缺口与建议）

### 2.1 URL 与 canonical 统一（P0）
现状：部分静态页面的 canonical 使用无扩展名路径（如 `/about`），但实际资源是 `about.html` 等，容易造成重复收录与权重分散。

建议：二选一并全站统一（含站内链接、sitemap、站内跳转逻辑）
- 方案 A：全部使用 `.html` 作为规范 URL（最符合纯静态托管现状）。
- 方案 B：托管侧配置无扩展名重写（需要服务器/平台规则），并让所有链接切换到无扩展名。

同时统一：是否保留末尾 `/`、参数顺序、大小写规则（只留一种）。

### 2.2 “每个游戏一个静态可索引页面”（P0）
现状：`game-detail.html` 依赖 JS 在客户端改写 `title/description/canonical/og`。搜索引擎可能执行 JS，但不稳定，且容易产生大量参数组合页。

建议：为每个游戏生成静态页（推荐）
- `game/<slug>/index.html`（推荐，便于扩展语言/资源）
- 或 `game/<slug>.html`（实现更简单）

保留兼容入口时：
- `game-detail.html?slug=...` 仅作为跳转页，301 到规范页；或设置 `noindex,follow` 并提供显式 canonical。

### 2.3 Sitemap 与 robots（P0）
- sitemap 只放规范（canonical）URL，避免把大量参数页放进去。
- URL 数量很大时：用 `sitemap-index.xml` + 分片（如 `sitemap-games-1.xml`、`sitemap-pages.xml`），并保证 `lastmod` 可维护。
- `robots.txt` 保持简单即可；若未来出现大量不可索引参数页，再有针对性 `Disallow`（谨慎避免误伤）。

### 2.4 结构化数据（Schema.org）（P0/P1）
站点级：
- `WebSite` + `SearchAction`：`target` 必须是站内真实可访问的搜索 URL（确保与实际搜索页一致）。
- `Organization`：不要填写不存在的电话/社媒链接；缺就删除或替换为真实信息。

游戏页：
- `VideoGame`（或 `Game`）：`name`、`description`、`genre`、`operatingSystem`（Web）、`applicationCategory`、`image`、`url`、（如可靠）`datePublished`。
- 如展示评分/热度：只有在页面可见且来源可信时才输出 `aggregateRating`，避免“编造评分”导致风险。
- `BreadcrumbList`：帮助搜索引擎理解层级结构。

### 2.5 语言与地区（按实际需要）
- 多语言站点用 `hreflang`，并确保每个语言版本都有对应可索引页面（不要只靠运行时翻译）。
- `html lang` 与页面语言一致。

## 3. 内容与长尾 SEO（不靠社媒也能增长）（P1）

### 3.1 低成本高收益内容类型
- 专题集合页：按机制/主题/设备友好度组织（触屏友好、双人、本地同屏、短局等）。
- 榜单页：每周/月固定更新节奏（“本月最热门”“本周新上架”），增加抓取新鲜度。
- 玩法指南/技巧：对少量头部游戏写 300–800 字差异化内容，覆盖“how to / tips / strategy”等意图。
- FAQ/术语表：解释常见玩法机制与术语（match-3、idle、roguelike 等），覆盖教育型关键词。

### 3.2 内链策略（让站内“可发现”）
- 首页：强化到分类/专题/榜单的入口，不要只堆游戏卡片。
- 分类/标签页：分页可索引；每个聚合页补充 200–400 字简介（避免纯列表薄内容）。
- 游戏页：固定模块把用户带回分类/标签页，并提供相似游戏推荐，形成网状内链。

### 3.3 外链游戏的 SEO 处理
- 外链游戏页依然作为“内容落地页”存在（介绍/截图/玩法/相似推荐），把外链按钮作为 CTA。
- 对不可控外链页面：避免把站内大量权重只导出不回流，务必加强站内相似推荐与分类入口。

## 4. 性能与可访问性（SEO 的技术底盘）（P2）

### 4.1 Core Web Vitals / 性能
- 减少阻塞渲染资源与第三方脚本影响；非首屏脚本尽量 `defer`/延迟加载。
- 图片统一规格：提供 WebP/AVIF（如可行），补齐 `width/height` 降低 CLS。
- 控制字体字重数量，确保 `font-display: swap`，避免首屏白屏。
- 静态资源开启长缓存（如有构建流程可加指纹文件名）。

### 4.2 可访问性与语义化
- 每页唯一 `h1`，标题层级连续。
- 图片与关键按钮补齐有意义的 `alt`/可访问文本。
- 弹窗/筛选/分页支持键盘操作，并处理焦点回收与 ESC 关闭。

---

## 5. 监控与维护（最小但要到位）

- Google Search Console + Bing Webmaster Tools：提交 sitemap，关注覆盖率、抓取错误、重复 canonical、软 404。
- 自动化校验（本地脚本或 CI）：
  - sitemap 中 URL 是否 200、是否为 canonical、是否被 `noindex`。
  - 抽样检测页面 `title/description` 唯一性与长度（避免大量重复）。
  - 死链扫描（404）与重定向链路检查。

## 6. 最短路径落地清单（建议先做这 5 条）

1. 统一并修正所有静态页 canonical（`.html` 或无扩展名二选一）。
2. 增加“每个游戏一个静态页”的生成流程（从 `games-data.js` 生成 HTML 模板）。
3. sitemap 仅包含规范游戏页（必要时分片/索引），并维护 `lastmod`。
4. 游戏页补齐 `VideoGame` + `BreadcrumbList` 结构化数据（与页面可见内容一致）。
5. 补齐分类/标签页与 `404.html`，并强化内链（让每个游戏页可从多处被找到）。

## 7. 针对当前仓库的“立即可做”清单（更具体）

### 7.1 修正明显不一致项（P0）
- canonical：`about.html`、`contact.html`、`terms.html`、`privacy.html` 的 canonical 目前是无 `.html` 路径，建议统一为实际可访问的规范 URL（或配合托管重写后再统一切换）。
- `WebSite/SearchAction`：首页结构化数据里 `target` 是 `https://arcadebloom.com/search?q=...`，但站内目前更像是 `all-games.html?search=...`（且参数名为 `search`）。建议让 `target` 与真实搜索页一致，否则会降低结构化数据有效性。
- OG 图片路径：页面里引用 `https://arcadebloom.com/images/og-image.jpg`，但仓库里没有 `images/` 目录（已有 `pic/`）。建议补齐真实可访问的 OG 图并统一路径。
- `Organization` 结构化数据：如电话、社媒链接为占位/不存在，建议删除或改为真实信息，避免“可信度”问题。

### 7.2 参数页与重复收录（P0）
- 你们的 `sitemap.xml` 里包含大量 `game-detail.html?id=...&name=...&category=...&slug=...` 这类组合 URL。建议：
  - sitemap 只保留“规范 URL”（最好是 `game/<slug>/` 这种无参数的静态页）。
  - 如果短期内必须保留参数页：确保 canonical 指向同一个规范页，并考虑给参数页加 `noindex,follow`。

### 7.3 元信息的唯一性（P0/P1）
- 头部页面（首页/全量页/详情页）`title`/`description` 不要大量模板化重复；尤其是“游戏详情页”，建议每个游戏页独立输出唯一的 `title` 与描述。

## 8. 可执行的 SEO 自检清单（上线前/每次批量更新后跑一次）

- URL 规范：全站只有一种 URL 形态（是否带 `.html`、是否带尾 `/`、是否允许大写、参数排序）。
- canonical：每页只有一个 canonical，且 200 可访问，且与 sitemap 中一致。
- 索引控制：
  - 需要被收录的页：`index,follow`。
  - 过滤/排序/组合参数页：优先 `noindex,follow` 或直接不生成可抓取入口。
- 标题与描述：
  - `title` 50–60 字符左右且唯一。
  - `description` 120–160 字符左右且描述具体玩法/特色。
- 结构化数据：JSON-LD 可解析、字段真实、与页面可见内容一致。
- 图片：关键图片有 `alt`，并可被抓取；OG 图真实存在。
- 页面可用性：移动端可用（按钮/iframe/全屏）、无明显控制台错误。
- Core Web Vitals：LCP/CLS/INP 关注首屏与游戏 iframe/大图。

## 9. 生成与维护建议（让目录规模化不崩）

### 9.1 建议增加“静态生成”步骤（P0/P1）
从 `games-data.js` 自动生成：
- 每个游戏的静态页：`game/<slug>/index.html`（包含独立元信息、结构化数据、面包屑、相似推荐）。
- 分类页/标签页：按目录数据生成分页。
- `sitemap-index.xml` + 分片 sitemap（避免单文件过大）。

好处：
- 搜索引擎无需依赖 JS 执行即可拿到核心信息。
- sitemap 与 canonical 变得一致且可批量校验。
- 新增游戏只需改数据并重新生成。

### 9.2 Feed（可选但很实用）
- 提供 `rss.xml`（新上架/更新游戏），方便搜索引擎与用户订阅工具发现增量内容。

### 9.3 质量闸门（可选）
- 写一个简单脚本做静态检查：
  - 检查 `games-data.js` 的 `slug` 是否唯一、图片路径是否存在、描述长度是否达标。
  - 抽样生成页面并校验 `title/canonical/og` 是否为空。
